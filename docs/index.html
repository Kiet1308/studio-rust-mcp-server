[ 04 — ARCHITECTURE ]

How it reaches into a
live play session.

Every DataModel — edit, play-server, play-client — runs its own plugin instance and long-polls the Rust server with a role tag, so commands route to the right context. The client can't speak HTTP, so the server relays through a chunked RemoteEvent bridge.

role=edit
DataModel · Edit

Edit plugin

Toolbar, place edits, and the privileged image read-back. Long-polls over HTTP.

role=server
DataModel · Play Server

Server plugin

Runs server code and relays on the client's behalf — the client has no HttpService.

role=client
DataModel · Play Client

Client plugin

Where the player lives: virtual input, screenshots, PlayerGui, character.

// Real input, not fake events

Input uses UserInputService:CreateVirtualInput() — the engine's actual pipeline. SendKey, SendMouseButton, SendMousePosition & SendMouseDelta mean WASD walks the character and clicks fire .Activated.

// Pixels never cross the relay

Screenshots capture on the rendering DataModel via CaptureScreenshot(); the temp contentId is read back as pixels on the edit DataModel with CreateEditableImageAsync — so multi-MB image data never touches the bridge.

// Clicks are hit-tested first

GetGuiObjectsAtPosition + a ZIndex/DisplayOrder render-order check finds the element that would actually receive the click. A surprise popup fails the call by name instead of swallowing the input.

// Survives a minimized window

Studio pauses rendering when minimized or covered, silently dropping input. The server detects the "not rendering" state, restores the window at the OS level, and retries the command once.
